## 项目概述

### 我为什么要做这个项目

日常浏览国际新闻时发现两个痛点：

1. **信息过载** — 每天面对多个新闻源、几十上百条报道，很难快速抓住重点
2. **真假难辨** — 自媒体时代谣言泛滥，普通读者缺乏判断新闻可信度的参照系

于是我设计了一款工具：自动聚合多个权威新闻源，用 AI 做一句话摘要和可信度评分，让用户花 3 分钟就能掌握当日要闻，同时有一个可信度参考来辅助判断。

### 这个产品做了什么

```
多源 RSS → 自动抓取 → URL 去重 → Claude AI 摘要+评分 → 前端展示
```

核心体验链路：用户在浏览器打开链接 → 看到按可信度排序的新闻卡片 → 一眼看到分数和摘要 → 点击卡片跳转原文。

## 产品亮点（AI PM 视角）

### 1. AI 产品设计：不是"调 API"，是定义 AI 的行为规范

我为 Claude 设计了完整的评分体系（Prompt Engineering），包含：

- **5 维度评分标准**（来源权威性、信息具体程度、原始出处可追溯性、用词客观性、交叉验证可行性）
- **5 档评分区间**（90-100 / 70-89 / 50-69 / 30-49 / 0-29），每档有明确的行为描述
- **输出格式约束**（严格 JSON，40 字以内摘要，50 字以内评分理由）

这本质上是一份 **AI 行为规范文档**（类似产品 PRD 中的算法需求），定义了 AI 在这个产品中"应该怎么想、怎么输出"。

### 2. 容错设计：AI 不可靠时的降级策略

AI 可能返回非 JSON 格式、可能超时、可能报错。我设计了降级路径：

> AI 失败 → 回退到来源权重 × 60% 作为默认分 → 标记为"AI 处理失败"

**产品意义**：用户不会看到报错或空白，核心体验链路不中断。这是做 AI 产品的基本素养 — AI 是概率系统，永远需要 fallback。

### 3. 信息架构设计：减少认知负荷

| 设计决策 | 产品逻辑 |
|----------|----------|
| 卡片式布局 | 每条新闻独立成卡，视觉上可快速扫描 |
| 高/中/低三档 badge | 用颜色编码（绿/橙/红），一眼可辨可信度 |
| 评分理由仅低可信度显示 | 高可信的新闻不需要解释"为什么可信"，只有低可信才需要提醒用户注意 |
| 默认按可信度排序 | 用户打开第一眼看到的是最可靠的新闻 |
| 仅高可信开关 | 满足快速浏览场景，过滤掉噪音 |
| 整个卡片可点击 | 降低点击成本，不需要精确点到标题 |

### 4. 用户需求驱动的功能迭代（从 git 提交历史可见）

| 轮次 | 用户反馈 | 我的响应 |
|------|----------|----------|
| V1 | "信息太多太杂" | 引入 AI 摘要和可信度评分 |
| V2 | "评分理由每条都显示，看着累" | 改为仅低可信度显示 |
| V3 | "只能点标题跳转，不方便" | 整个卡片可点击 |
| V4 | "有些源需要梯子" | RSS 源全部替换为国内直连 |
| V5 | "画面两边太空" | 增加侧边栏（统计面板 + 信源列表） |
| V6 | "只能看预设源" | 增加自定义 RSS 源管理 |
| V7 | "排序只能降序" | 增加升降序切换 |
| V8 | "只能本地看" | 配置 GitHub Pages 自动部署，一个链接即可体验 |

### 5. 零摩擦的体验路径

面试官体验这个产品不需要：安装环境、注册账号、配置 API Key、等待加载。

只需一个 URL。这是有意为之的产品设计 — 降低体验门槛 = 提高触达率。

## 技术实现（证明能和工程师协作）

| 层级 | 技术 | 为什么选它 |
|------|------|-----------|
| AI 服务 | Claude API (Anthropic) | JSON 输出稳定、中文能力强 |
| 数据获取 | RSS 多源聚合 (rss-parser) | 零成本、实时、无版权风险 |
| 前端 | 原生 HTML/CSS/JS | 零依赖、加载快、证明基本功 |
| 数据持久化 | localStorage | 用户自定义源不丢失，无需后端 |
| 部署 | GitHub Pages + Actions | 免费、自动部署、全球 CDN |
